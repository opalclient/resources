# Publishing Scripts

A script can reach other people three different ways. These are independent
channels, not stages of one pipeline: pick the one that matches how public,
permanent, and vetted you want the script to be.

- **Private cloud sync** — your own scripts, following you between your own
  devices. Nobody else sees them.
- **In-client community gallery** — a lightweight, community-submitted
  browser inside the client itself.
- **The official gallery (GitHub)** — a single reviewed repository with CI
  and tagged releases. The recommended path for a script you want polished
  and broadly discoverable.

## Private Cloud Sync

Once a script is working locally, upload it to your account so it's there
when you switch devices:

```
.script upload MyScript
```

This uploads `scripts/MyScript.js` to your account. Nothing here is public;
it's a personal backup and sync mechanism, not a way to share a script with
anyone else.

<Callout type="info">
    Script names must be 1-32 characters. Max file size is 100KB.
</Callout>

### In-Game Commands

| Command | Description |
|---------|-------------|
| `.script upload <name>` | Upload local script to private cloud |
| `.script download <name>` | Download private script from cloud |
| `.script cloud list` | List your cloud scripts |
| `.script cloud delete <name>` | Delete a cloud script |

### Limits

| Limit | Value |
|-------|-------|
| Max file size | 100KB |
| Name length | 1-32 characters |
| Description | Max 256 characters |
| Version string | Max 16 characters |

## In-Client Community Gallery

A separate, lighter-weight surface lives inside the client itself: open it
from the command palette to browse scripts other users have submitted, vote
on them, and clone one straight into your own collection. There's no
maintainer review gate here beyond the submission itself, so quality and
safety vary from script to script — it's the fast path, not the vetted one.

<Callout type="warning">
    A script you clone from the community gallery is still a downloaded
    script from someone else. It lands in `opal/scripts/pending/` and needs
    an explicit **Trust & run** before it executes, the same as anything
    pulled from the GitHub gallery below. See [The Sandbox](sandbox).
</Callout>

## The Official Gallery (GitHub)

The curated, reviewed channel is a single GitHub repository:
[`opalclient/scripts`](https://github.com/opalclient/scripts). Every script
in it went through a normal pull request, got built and checked by CI before
a maintainer ever read it, and shipped as a tagged release with the bundled
`.js` attached. This is the channel to use when you want a script actually
reviewed and easy for anyone to find, rather than just floating in the
in-client community list.

### How It Works

```
Fork & copy template/ → Write manifest + src → Build & test locally → Open a PR
→ CI checks the submission → Maintainer review → Merge = approved, release tagged
```

1. **Fork the repo and copy the template.** `template/` is a working
   TypeScript scaffold. Copy it into a new folder named for your script's id.
2. **Write your script.** Fill in `manifest.json` and the source under `src/`.
   An existing, released script folder is the best reference for the shape
   maintainers expect.
3. **Build and test locally.** The template bundles your TypeScript down to a
   single `.js` with esbuild. Drop that built file into your own
   `opal/scripts/` folder and run `.script reload` to confirm it actually
   works before you open a PR.
4. **Open a pull request** against `opalclient/scripts`.
5. **CI runs automatically.** See [CI Gates](#ci-gates) below. A failing check
   blocks review.
6. **A maintainer reviews the code** once CI is green.
7. **Merge is the approval.** There's no separate sign-off step after that. A
   release tagged `<id>@<version>` is cut with your bundled `.js` attached,
   and that release is what the gallery serves to users.

### The Template

`template/` is a TypeScript project, not a bare `.js` file. You write your
module in TypeScript against the same proxy API these docs describe, and the
template's esbuild config bundles it into the single `.js` file Opal actually
loads.

<Callout type="info">
    Bundling to one file matters because the sandbox has no `require` or
    dynamic `import`: anything your script depends on has to be inlined at
    build time. If your local build produces more than one output file,
    something is wrong with the bundler config, not the sandbox.
</Callout>

### manifest.json

Every script folder needs a `manifest.json` alongside its `src/`. At minimum
it carries the script's id (the same id that shows up in the release tag) and
a version. Start from the template's `manifest.json`; CI's manifest-validation
gate tells you immediately if a required field is missing or malformed.

<Callout type="warning">
    This repo `manifest.json` is a different file from the `// ==OpalScript==`
    header you can put at the top of the script's own source. `manifest.json`
    is submission metadata that CI and maintainers read before your code ever
    ships anywhere; the `// ==OpalScript==` block is metadata the running
    client parses out of the file itself, at load time. Keep both, and keep
    them saying the same thing. See [Script Manifest](manifest) for the
    in-script header.
</Callout>

### CI Gates

Every pull request runs the same automated checks before a maintainer looks
at it:

| Gate | What it checks |
| :--- | :--- |
| Manifest validation | `manifest.json` is well-formed and has the required fields. |
| Lint | Code style matches the repo's lint config. |
| Typecheck | The TypeScript compiles clean. |
| Build size | The bundled `.js` comes in under 1 MB. |
| Tests | The script's own tests pass. |

<Callout type="warning">
    A failing CI check blocks review outright. Fix it and push again; there's
    no manual override.
</Callout>

### Review

Once CI is green, a maintainer reads the actual code for three things:
**functionality** (does it do what the description says), **safety** (no
credential theft, no malicious behavior, no excessive resource use), and
**quality** (readable, focused code with a clear purpose). Merging the PR
*is* the approval — there's no separate sign-off afterward.

<Callout type="info">
    Keep obfuscation out of your submission. A reviewer who can't read your
    code can't approve it, sandboxed or not.
</Callout>

### Releases

A merged PR is cut as a release named `<id>@<version>`, with the bundled
`.js` attached as the release asset. That tag and that file are what actually
ship to users, so treat `manifest.json`'s version field as real metadata, not
decoration.

#### Updating a Published Script

Bump the version in `manifest.json`, rebuild, and open a new PR against your
existing script folder. There's no fast path around review for an update: it
runs through the same CI gates and the same maintainer review as a first
submission, and a merge cuts a new release tagged with the new version.

### Tips for Getting Merged

1. **Test locally first.** Load the built `.js` from your own
   `opal/scripts/` folder and run it before opening a PR — CI catches build
   failures and lint issues, not whether the module actually behaves.
2. **Keep it focused.** One script, one purpose.
3. **No obfuscation.** Reviewers need to be able to read your source.
4. **Document your settings.** Clear names for `addBool`, `addNumber`, and
   `addMode` settings save the reviewer, and every future user, a guess.
5. **Version meaningfully.** Start at `1.0.0` and bump it for every change
   you ship.
6. **Look at a real example.** Chomp is the repo's flagship script and a
   working, `storage`-using reference for exactly this pipeline. See
   [Storage](core/storage) for the persistence side of it.
