# Scripting Introduction

Opal's scripting engine runs on **GraalVM JavaScript (ES6+)**, hot-reloaded from the `opal/scripts/` folder.

---

<Callout type="info">
    **Scripts run sandboxed.** Every `.js` file is evaluated in a GraalVM context locked to the proxy API documented here: no filesystem access, no thread creation, and no reflective reach into the Minecraft client or Opal internals beyond the methods each proxy explicitly exposes. A script cannot read or write files, spawn a thread, or call into client code that isn't part of the documented API surface.
</Callout>

The proxy API below is the *complete* surface a script can reach. Anything not exposed by a proxy, a bound type, or an event payload is unreachable from script code, not merely discouraged.

### Community scripts are quarantined

Scripts imported from the public gallery ([`opalclient/scripts`](https://github.com/opalclient/scripts)) never auto-run. They are written to a **quarantine directory** (`opal/scripts/pending/`), which the loader skips entirely: a pending script does not execute on reload or restart. It runs only after you explicitly **Trust & run** it, at which point it is moved into `opal/scripts/` and loaded. Your explicit trust is the gate that first executes community code; review it first.

## What a Script Can Do

A single script can register any combination of:

- **Modules**: feature toggles that appear in the ClickGUI alongside native modules, with their own settings and event handlers.
- **HUD islands**: custom Dynamic Island overlays drawn on the HUD via the `overlay` global.
- **Command-palette views**: interactive mini-apps and games inside the command palette via the `palette` global. See [Palette Views](ui/palette).

## Anatomy of a Script

Every script entry point must register itself using `registerScript`. This returns a handle used to register a module.

```javascript
const script = registerScript({
    name: "My Script",
    version: "1.0.0",
    authors: ["User"]
});

script.registerModule({
    name: "AutoGreeter",
    description: "Automatically greets players"
}, (module) => {
    module.on("enable", () => {
        client.print("Module enabled!");
    });

    module.on("disable", () => {
        client.print("Module disabled!");
    });
});
```

Save it as a `.js` file in your `opal/scripts/` folder, then run `.script reload` in-game to load it. Your module appears in the ClickGUI alongside native modules.

## Module Settings

Scripts can define settings that appear in the ClickGUI identically to native module settings. All settings must be defined inside the `registerModule` callback, before any `module.on(...)` calls.

```javascript
script.registerModule({ name: "MyModule", description: "..." }, (module) => {
    module.addBool("Enabled", true);
    module.addNumber("Speed", 1.0, 0.1, 5.0, 0.1);
    module.addMode("Mode", ["Vanilla", "NCP"]);
    module.addGroup("Options", ["Speed", "Mode"]);

    module.on("preGameTick", () => {
        if (!module.getBool("Enabled")) return;
        const speed = module.getNumber("Speed");
        const mode = module.getMode("Mode");
    });
});
```

### Setting Types

**Boolean (Toggle)**
```javascript
module.addBool("Auto Sprint", true);
module.getBool("Auto Sprint");          // true or false
module.setBool("Auto Sprint", false);
```

**Number (Slider)**
```javascript
module.addNumber("Reach", 3.1, 3.0, 6.0, 0.1);
//                name    default min  max  step
module.getNumber("Reach");              // 3.1
module.setNumber("Reach", 3.3);
```

**Mode (Dropdown)**
```javascript
module.addMode("Mode", ["Silent", "Normal", "Fast"]);
module.getMode("Mode");                 // "Silent"
module.isModeEqual("Mode", "silent");   // true (case-insensitive)
```

**Group (Collapsible)**
```javascript
module.addBool("Show ESP", true);
module.addNumber("ESP Range", 64, 1, 128, 1);
module.addGroup("ESP Settings", ["Show ESP", "ESP Range"]);
```

<Callout type="warning">
    Settings must be defined before `module.on(...)` calls. `addGroup` must be called after all settings it references have been defined.
</Callout>

Settings are **automatically saved** to your config. They persist across restarts and reloads.

## Working with Numbers and Colors

JavaScript has a single numeric type: the 64-bit `double`. There is no integer type. This matters when you pass values into the Java API.

Colors are packed **32-bit ARGB integers**. It is tempting to write a literal like `0xFFFF0000`, but that value (`4294901760`) is larger than the maximum signed 32-bit integer (`2147483647`). When that `double` is narrowed back to a Java `int`, the high (alpha) bit overflows and the color comes out wrong.

<Callout type="warning">
    Build colors with `renderer.color(r, g, b[, a])` (components range from 0 to 255, alpha defaults to `255`) rather than raw `0xAARRGGBB` literals. A fully opaque white is `renderer.color(255, 255, 255)`, not `0xFFFFFFFF`.
</Callout>

```javascript
const red   = renderer.color(255, 0, 0);          // opaque red
const panel = renderer.color(20, 20, 28, 120);     // translucent dark panel
```

## Java Interop

Specific Java classes and constants are exposed globally. You do not need to import these.

| Global | Description |
| :--- | :--- |
| `Vec3d` | World vector (wrapped as `ScriptVec3`, readable `getX/getY/getZ`, plus vector math) |
| `Vec2f` | Yaw/pitch rotation pair (wrapped as `ScriptVec2f`, readable `getYaw/getPitch`) |
| `BlockPos` | Block position (wrapped as `ScriptBlockPos`, readable `getX/getY/getZ/offset`) |
| `Color` | `java.awt.Color`; its constructors and `getRGB()` are allow-listed |
| `MAIN_HAND` / `OFF_HAND` | `InteractionHand` constants; pass-through tokens for interaction calls |
| `keys` | Key codes for binds and palette-view key handlers (e.g. `keys.F7`) |

<Callout type="warning">
    **Call getters; never read them as properties.** The sandbox does no bean-property mapping, so `mc.player` is `undefined`, not the player, and `if (mc.player === null)` is a guard that never fires. Write `mc.getPlayer() === null`. The same rule governs every wrapper: `vec.getX()`, never `vec.x`.
</Callout>

<Callout type="info">
    `Java.type(...)` is **not available**, and that is deliberate and permanent: the sandbox refuses every script-initiated host class lookup. The globals above are the curated set of classes a script gets, and that closed surface is what makes the public script gallery safe to offer. Use the proxy globals (`player`, `world`, `inventory`, etc.) for all client access.
</Callout>

## Available Globals

| Global | Description |
|--------|-------------|
| `client` | Chat output, module access, screen info, theme colors |
| `player` | Position, health, combat, movement state |
| `movement` | Speed manipulation, direction calculations |
| `rotation` | Rotation math, smooth aiming, FOV checks |
| `world` | Blocks, entities, time, dimension |
| `inventory` | Slots, items, searching, silent switching |
| `renderer` | 2D shapes, text, gradients, paths, images, and composite effects (shadows, blur fills, glows) |
| `overlay` | Dynamic Island HUD creation |
| `esp` | 3D-to-2D projection for ESP rendering |
| `modules` | Query, toggle, and list all modules |
| `notification` | Toast notifications (success, error, warn, info) |
| `palette` | Custom command-palette views (mini-apps, games) |
| `timer` | Millisecond timers for rate-limiting and delays |
| `storage` | Persistent per-script key-value storage, survives restarts |
| `mc` | Null guards for player/world, plus `interactionManager` |

See the **API Reference** pages for full method documentation.

## Commands

| Command | Description |
|---------|-------------|
| `.script reload` | Reload all scripts from disk |
| `.script list` | Show loaded scripts and their modules |
| `.script info <name>` | View script details (version, authors, settings) |
| `.script enable <name>` | Enable a script module |
| `.script disable <name>` | Disable a script module |
| `.script upload <name>` | Upload local script to your private cloud |
| `.script download <name>` | Download a private script from cloud |
| `.script cloud list` | List your cloud scripts |
| `.script cloud delete <name>` | Delete a cloud script |

`.script reload` performs a clean cycle: every loaded module's `disable` callback fires while its context is still alive, all contexts are closed, and then the `opal/scripts/` folder is rescanned from disk, so you don't need to restart the game.

## Tips

- Community scripts stay in `scripts/pending` until you Trust & run them, even though every script runs sandboxed to the proxy API.
- Guard with `if (mc.getPlayer() === null) return;` before touching player data, since it's null in menus and loading screens. Call the getter: `mc.player` is `undefined`, so that guard silently never fires.
- Collections come back as a read-only `ScriptList`: iterate with `for..of`, index with `list[i]`, or read `.length`, and `size()`/`get(i)` still work. `.map`/`.filter` are absent; spread into an array first (`[...list].map(...)`).
- Use `renderScreen` for HUD drawing; renderer methods only work during render events.
- Use `preGameTick` for logic that needs to run every tick (20 times a second).
- Build colors with `renderer.color(...)` instead of raw `0xAARRGGBB` literals, since JS numbers narrow to `int` incorrectly.
- Errors report where they came from (`MyScript.js:42: ...`), and a handler that overruns a ~250ms budget is stopped, so a runaway loop can't freeze the game. See [The Sandbox](sandbox).
- Print-debug with readable forms: `console.log(vec)` or `client.print(vec)` shows `ScriptVec3(...)` rather than `JavaObject[...]`.
- Declare an optional `// ==OpalScript==` header for name, version, and author metadata. See [Script Manifest](manifest).
- Each script is isolated in its own context, so scripts can't interfere with each other.
- Hot reload with `.script reload`; no restart needed.
- Max script size: 100KB.

## Editor and AI Tooling

A few tools exist to make writing scripts faster, independent of this documentation:

- **[Example scripts](https://github.com/opalclient/opal-scripts)**: a gallery of runnable, documented example scripts covering every proxy category, from a simple toast notification to full palette-view mini-games.
- **[VS Code and JetBrains extensions](https://github.com/opalclient/opal-scripting-ide)**: ambient type definitions for the entire proxy API, so hovering or autocompleting `player.`, `renderer.`, `world.`, etc. shows real signatures and doc text in a plain `.js` file, plus a one-command script scaffold and boilerplate snippets/live templates. Prebuilt plugins are attached to the repo's [GitHub Releases](https://github.com/opalclient/opal-scripting-ide/releases): install the VS Code `.vsix` with **Install from VSIX...**, or the JetBrains `.zip` with **Plugins → Install Plugin from Disk...**.
- **[AI assistant setup](https://github.com/opalclient/opal-script-skills)**: a portable skill package that teaches Claude Code, Codex, Copilot, Gemini, and similar tools the full scripting API, installed with `npx @opalclient/opal-script-skills add`, plus a `/new-opal-script` command for Claude Code that scaffolds a working script from a prompt.

These are separate, independently versioned open-source projects.
