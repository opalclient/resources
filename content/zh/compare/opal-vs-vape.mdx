## 每个人是如何进入你的游戏

这是它们之间最大的架构差异：Opal 是一个 Fabric 模组，Vape 是一个外部 DLL 注入器。

Opal 通过 Mixin 作为 Fabric 模组加载，注册了大约 74 个针对 MC 26.2 的 Mixin 类，在整个堆栈中没有 DLL 注入。Vape 自身的条目将其注入模型列为外部：从 vape.gg 加载的 DLL 进入 javaw.exe 进程。这种加载方式正是导致 Vape 在至少一个公开屏幕共享指南中被字符串匹配识别的原因，工作人员会扫描 javaw.exe 内存，寻找字面字符串“vape”，并与显示相同 DLL 的预取条目进行核对。这并非声称 Fabric 模组是不可见的，而仅仅是这两个客户端物理连接到 Minecraft 的方式上的一个真实且有记录的差异，值得单独权衡，而不是一带而过。

## 脚本和自定义

Opal 包含一个完整的 JavaScript 脚本引擎。“Vape”的脚本故事也未在我们记录中。

Opal 将GraalJS嵌入为GraalVM的多语言宿主，并提供代理类（PlayerProxy、RenderProxy、RotationProxy、WorldProxy、ModuleProxy），这些代理类将内部客户端 API 暴露给用户脚本，可在运行时加载和卸载。它位于基于相同 Blaze3D 效果引擎的GraalJS/Yoga UI 运行时中。我们没有关于“Vape”是否提供任何类似功能的文档，因此我们不愿猜测，只是说：我们不知道，你也应该避免任何假设。

## 战斗调整

Opal 记录了完整的战斗和移动模块集，并按名称进行记录。“Vape”的文档不够完善，无法进行直接比较.

Opal的战斗配置包括KillAura，带有独立的选定目标和追踪瞄准子包，Scaffold，Flight，LongJump，Speed，NoSlow，一个ESP家族，以及Blink用于重新安排数据包序列，这些都针对BedWars/SkyWars风格的小游戏战斗和实用性玩法进行了优化。Vape的条目没有指定任何游戏风格或服务器重点。唯一记录的部署细节是它在服务器上广泛使用，而且没有明确的战斗优化方向。
